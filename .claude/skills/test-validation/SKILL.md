# test-validation

生成済みテストの**妥当性**を評価するスキル。cc-sddは実装と同時にテストを作るが、そのテストが本当に回帰を検出できるかは別問題。ここを機械 + 目視の両方で検証する。

## 大原則: カバレッジ数値 ≠ テストの妥当性

カバレッジが測るのは「テスト実行時にそのコードを**通過したか**」だけで、「その挙動が**検証されたか**」ではない。カバレッジ100%でも以下は全て起こり得る。

- ハッピーパスしか通さない（異常系・境界値が抜ける）
- アサーションが弱い（`status_code == 200` だけでボディを見ない、`is not None` で終わる）
- モックしすぎて実質何もテストしていない（テスト対象そのものをモックしている）
- 実装をなぞっただけのトートロジー（ロジックが間違っていてもテストが通る）

**このスキルの中心は「もしこのロジックが間違っていたら、このテストは落ちるか?」を検証すること。** カバレッジは下地に過ぎない。

## 評価の優先順位

層によって「良いテスト」の姿と、掛けるべき労力が違う。上から順に手を入れる。

| 優先 | 層 | 方針 | カバレッジ目標 | 主な手段 |
|---|---|---|---|---|
| ◎ 最優先 | ビジネスロジック層（Service / UseCase） | 分岐・異常系・ビジネスルールを厳密に | 90%以上 | mutation testing + 統合テスト（ROLLBACK分離） |
| ✅ 高 | データ変換・バリデーション層（Validator / Transformer） | valid/invalid/境界を網羅 | 85%以上 | parametrizeテーブル駆動 |
| ✅ 中 | ユーティリティ関数（utils/） | エッジケース中心 | 80%以上 | parametrize |
| ❌ 原則不要 | 単純なgetter・純粋なDTO | スキップ | — | — |

全体は 80% を基準とする。ただしこの目標値は**トリアージ用**であり、達成＝妥当ではない。数値を満たしていても mutation testing や必須パターンで空洞と判定されることがある（3ゲートの関係は次節）。

**重要な例外**: validator や `@computed_field` を持つ Pydantic モデルは「単純なDTO」ではない。ロジックを持つ以上、Validator層と同じ扱いで境界値テストの価値がある。純粋にフィールドを持つだけのものだけスキップする。

### カバレッジ計測から除外する対象

以下はテスト不要。カバレッジの分母から外して、意味のある数値だけを見る。

- 設定ファイル（`config/`）
- 型定義のみのファイル（TS: `types/` / `*.d.ts`、Python: 型スタブ）
- 単純な re-export（TS: `index.ts`、Python: 純粋な `__init__.py`）
- マイグレーションファイル（Alembic の `versions/`、フロントのDBマイグレーション）
- 純粋DTO・単純getter（前表の❌層）

## 妥当性を測る3つのゲート

このスキルは性質の違う3つのゲートでテストを評価する。1つでも通らなければ補強対象。

1. **カバレッジ閾値**（トリアージ）— そもそもコードに触れているか。低い箇所から優先的に見る。Step 2。
2. **必須5パターン**（完全性）— 必要なシナリオを網羅しているか。Step 4。`references/review-checklist.md` 参照。
3. **mutation / “落ちるか”の一問**（深さ）— アサーションが回帰を検出できるか。Step 3・Step 4。

**カバレッジ90%かつ5パターン網羅でも、アサーションが弱ければ 3 で空洞と判定される。** 数値・網羅・深さは別物。

## 評価の単位（src と tests の対応）

評価は「テストファイル」と「対象実装」の**ペア**で行う。両者は測る手段が違う。

```
src/schemas/color_master.py  ←→  tests/unit/test_color_schema.py
（カバレッジ・mutationで測る）      （中身を読んでチェックリスト＋インベントリ）
```

- **カバレッジ / mutation testing** はソース実装に対してしか測れない（テストファイル自体のカバレッジは無意味）。数値の測定先は常に**対象実装**。
- **テストインベントリ / レビューチェックリスト** はテストファイルを開いて評価する。
- **レポートのテーブル** はソースのモジュール名で並べる。

「tests/ をファイルごとに評価する」感覚で始めてよいが、各テストファイルから対応する実装を逆引きし、数値はその実装で測る、という関係を保つ。実装ファイルが手元に無い場合は、カバレッジ・mutation（Step 2-3）は測れないので、インベントリとチェックリスト（Step 0・Step 4）だけで評価し、その旨を明記する。

## 評価フロー

対象が渡されたら（モジュール名 / ディレクトリ / テストファイル / 「全体を評価して」等）、次の順に進める。

### Step 0. テストインベントリ（現状把握）

まず既存テストを**読んで**、「今このテストが何を・どこまで検証しているか」を一覧化する。妥当性を論じる前に現状を可視化する工程。テストファイルを開き、テスト関数ごとに次を記録する。

- 対象実装（逆引き）と推定層
- テスト関数名と、それぞれが検証している観点
- クラス/関数が複数なら、それごとにグルーピング
- **今カバーしている観点** と **見当たらない観点**（正常系だけか、invalid・境界・null はあるか）

これがそのままレポートの前半（現状把握）になり、Step 4 の欠落抽出の入口にもなる。

### Step 1. 対象の層を特定する

Step 0 で逆引きした対象実装が、優先度表のどの層に属するかを判定する。複数モジュールなら層ごとに仕分ける。DTO・単純getterしかないなら「評価不要」と明言して労力を割かない。

### Step 2. カバレッジのベースラインを取る

branch coverage を**必ず**有効にする。line だけだと条件分岐の片側が丸ごと抜けても検出できない。

バックエンド（pytest）:
```bash
pytest --cov=app --cov-branch --cov-report=term-missing --cov-report=html
```

フロントエンド（Vitest, v8 provider）:
```bash
npx vitest run --coverage
```

DTO・単純modelは数値から除外して、意味のあるカバレッジだけを見る。`pyproject.toml`:
```toml
[tool.coverage.run]
branch = true
omit = [
  "app/schemas/*",   # 純粋なPydantic DTOのみ。validator付きはここに入れない
]
```

`term-missing` の **未通過行（Missing列）と未通過分岐** を記録する。あわせて機械可読な `coverage-summary.json`（Vitest）/ `coverage.json`（pytest `--cov-report=json`）を参照し、ファイルごとに次の基準で分類する。

- ❌ **50%未満** → 即対応
- ⚠️ **50〜80%** → 優先度高
- ✅ **80%以上** → 維持でOK（ただし数値だけで妥当とはしない。Step 3-4 へ）

分類結果を優先度表（層と目標値）と突き合わせ、「◎層なのに⚠️」のような優先度と実績のギャップが大きいものから着手する。カバレッジ数値そのものは合否判定にしない — あくまでトリアージの下地。

### Step 3. mutation testing で自動評価（◎層に集中）

これがカバレッジでは絶対に見えない核心。コードにわざとバグ（`+`→`-`、`>`→`>=`、条件反転など）を仕込み、テストがそれを**落とせるか**を測る。ミューテーションを入れてもテストが通る = そのテストは回帰を検出できない空洞、と機械的に判定できる。

全コードに掛けると重いので **priority ◎ の Service/UseCase 層に絞る**。ここは分析より「実際に走らせて survived を出す」ことが価値。**必ず一度は実行する**（時間箱で省略しない）。

**第一選択＝同梱ランナー `scripts/mutation_runner.py`（依存ゼロ・要インストールなし）。** DB 非依存で速く回る対象（純粋ロジック・Validator・utils・切り出した◎ロジック）に使う。外部ツール導入不要で Windows でも完走し、`.coverage` ロックや mutmut の環境差を回避する。

```bash
# mutation（1 ソースへ変異 → survived を列挙。project-dir は pytest を回す位置＝backend/）
python <skill>/scripts/mutation_runner.py mutate \
    --source src/services/metrics.py --tests tests/unit/test_metrics.py --project-dir backend

# branch coverage（Windows の .coverage ロックを退避して term-missing）
python <skill>/scripts/mutation_runner.py cov \
    --module src.services.metrics --tests tests/unit/test_metrics.py --project-dir backend
```

注意: 対象は **DB 非依存で速いテスト**に絞る（統合テストは1変異ごとに遅く非現実的）。DB が絡む◎ロジックは、判定部を純関数として切り出してから掛けると効く（例 `metrics.compute_rates`）。より大規模・本格運用は mutmut も可。セットアップ・結果の読み方・mutmut は `references/mutation-testing.md` を参照。

生き残ったミューテーション（survived mutants）が「弱いテスト」の具体的リスト。これを潰す形でテストを補強すれば、「カバレッジを上げる」作業が自動的に「意味のあるテストを書く」作業になる。

**重要（mutation score を過信しない）**: score 100% / survived 0 は「**ツールが生成した種類の変異を検出できた**」だけで、テストが健全である証明ではない。演算子・定数変異は次を**構造的に見つけられない**。クリーンな結果が出ても、必ず Step4 のチェックリスト（特に下記）を当てること。

- **同型の出力どうしの取り違え**（例: `false_alarm_rate` と `miss_rate` の代入を入れ替える）。演算子変異では生成されない。
- **フィクスチャの値衝突でマスクされる欠落**（例: テストが `fp_num=1, miss_num=1` と同値だと、上の取り違えが起きても両方同じ値になり検出できない）。→「フィクスチャは、入れ替え・取り違えが起きたら結果が変わる**相異なる値**を使っているか?」を必ず問う。
- **同値クラスの抜け**（例: `annotated>0 かつ fp=0` のとき率は NULL でなく 0.0。このケースが無いと『0 のとき None を返す』変異が生存する）。

survived 0 は「無理に欠落を作らない」根拠にはなるが、それは**上記の意味論チェックを通した後**の話。ツール結果だけで「健全」と結論しない。

### Step 4. レビューチェックリストで目視評価

mutation testing は挙動は見るが「テストの意図が正しいか」までは見ない。層ごとの観点でテストを棚卸しして空洞を炙り出す。詳細な観点と層別の「良いテストの姿」は `references/review-checklist.md` を参照。

**必須5パターンの完全性チェック** — 各テスト対象について、次の5パターンが揃っているか確認する。欠けているパターンがあれば「テスト生成は不完全」とみなし、欠落として報告する。

1. 正常系（正しい入力で期待通りの結果）
2. 異常系・入力不正（null・undefined・空文字・型違い）
3. 境界値（最小・最大・ちょうど上限・上限+1）
4. 副作用の検証（DB書き込み・外部API呼び出し回数 など。※副作用を持つ対象のみ）
5. 例外ハンドリング（エラーが正しく throw / raise されるか）

パターン4は副作用のある対象にのみ適用する（純粋関数・純粋Validatorには副作用が無いので対象外）。それ以外の4パターンは全層で必須。

最重要の一問（どの層でも必ず問う）:
> **そのロジックが間違っていたら、このテストは落ちるか?**

あわせて、単体で検証している validation/ガードが**本番の実行経路に配線されているか**を一度たどる（schema の `Literal` を単体テストしていても、API が生の `str` を受けていれば無効値は本番で弾けず、そのテストの保証は宙に浮く）。手順化されたカバレッジ・インベントリ作業に集中すると見落としやすい観点。

### Step 5. レポートを出す

評価結果は下記の形式で必ずまとめる。数値の羅列ではなく「次に何をすべきか」を優先度付きで示す。

```
## テスト妥当性評価: <対象>

### テストインベントリ（現状把握）
- 対象実装: <逆引きしたsrcファイル>（推定層: <◎/✅/❌>）
- テスト関数: <N>件
  - <関数名> — <検証している観点>
  - ...
- 今カバーしている観点: <箇条書き>
- 見当たらない観点: <箇条書き。invalid/境界/null など>

### サマリ
- 評価対象: <モジュール/層>
- 総合判定: <健全 / 補強推奨 / 空洞リスク高>

### 層別の状況
| モジュール | 層 | cov（分類） | 5パターン | survived mutants | チェックリスト所見 |
|---|---|---|---|---|---|
| ... | ◎Service | 82% ✅ | 4/5（副作用欠） | 5件 | 異常系欠落 |

（cov分類は ❌<50% / ⚠️50-80% / ✅80%+。5パターンは正常・異常・境界・副作用・例外の充足数。実装が手元に無くカバレッジ/mutationを測れない場合は、その列を「未計測」とし理由を明記）

### 検出した空洞・欠落（優先度順）
1. [◎] <module>: <具体的な欠落。例: 在庫0件時の分岐にテストなし。mutant `qty > 0`→`qty >= 0` が生存>
2. ...

### 推奨アクション
- <生き残ったmutantを潰すための具体的なテスト案>
```

### Step 6. 補強（依頼された場合）

生き残ったmutantを潰す / チェックリストの欠落 / 不足している必須パターンを埋める形でテストを追加する。追加したテストは必ず「一度落ちること」を確認する（該当ロジックを一時的に壊して赤→直して緑）。落ちないテストは空洞。

**テストが書きにくいコードへのアプローチ** — 補強時に「テストしにくい」と感じたら、多くは設計のシグナル。次の方針で対処する。

- **外部依存が多い**: 外部サービス・DBはモック化してユニットテストを書く。インターフェースが切れているならそこでモックを差し込む。切れていないなら、その結合こそ設計上の問題として指摘する。
- **副作用が混在**: 副作用のある処理を切り出してから個別にテストする。切り出しが難しければスパイで呼び出し（引数・回数）だけ検証する。
- **対象が巨大**: 関数の責務を確認し、最も影響範囲の広い critical path から順に対応する。全パスを一度に埋めようとしない。

## 層別テストの原則（要約）

- **Service/UseCase（◎）**: mutation testing の効果が最大。モックは FTP・WebSocket・外部I/O だけに留め、ビジネスルールは実物のロジックを通す。DBが絡む部分は transactional ROLLBACK 分離を使った統合テストで。Service が DB フル構築なしにテストしにくいなら、それ自体が結合度の高さのシグナルで設計を疑う余地がある。
- **Validator/Transformer（✅）**: `@pytest.mark.parametrize` のテーブル駆動が最適。valid/invalid/境界を一覧で並べる。生成テストが正常系だけなら真っ先に補強対象。
- **utils（✅）**: 同じくparametrizeでエッジケース中心。
- **getter/純粋DTO（❌）**: スキップ。ただしvalidator/computed_field持ちは上のValidator層扱い。

## CIへの組み込み

整備済みのGitHub Actions CIに乗せる場合、**全体一律のカバレッジゲートは空洞テストを量産する誘因になる**ので避ける。Service層など優先層に限定した閾値、または mutation score（mutmutの生存率）を◎層だけで監視する方が健全。詳細は `references/mutation-testing.md` の「CI連携」を参照。
