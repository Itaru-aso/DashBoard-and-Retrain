# レビューチェックリスト（空洞テスト検出）

mutation testing が「挙動を検出できるか」を測るのに対し、このチェックリストは「テストの意図が正しいか・観点が抜けていないか」を目視で確認するためのもの。生成テストを1ファイルずつ、または層単位で棚卸しする。

## 必須5パターンの完全性チェック

各テスト対象について、次の5パターンが揃っているかを最初に確認する。**欠けているパターンがあれば「テスト生成は不完全」とみなし、欠落として報告する。**

| パターン種別 | 具体例 | 適用範囲 |
|---|---|---|
| ① 正常系 | 正しい入力で期待通りの結果が返る | 全層 |
| ② 異常系（入力不正） | null・undefined・空文字・型違い | 全層 |
| ③ 境界値 | 最小値・最大値・ちょうど上限・上限+1 | 全層 |
| ④ 副作用の検証 | DBへの書き込み・外部API呼び出し回数 | 副作用を持つ対象のみ |
| ⑤ 例外ハンドリング | エラーが正しく throw / raise されるか | 全層 |

④は純粋関数・純粋Validatorには副作用が無いため対象外。それ以外の4つは全層で必須。レポートには「5パターン: 3/5（境界・副作用が欠落）」の形で充足数を記す。

## 全層共通の最重要チェック

まず全てのテストにこの問いを当てる。

- [ ] **そのロジックが間違っていたら、このテストは落ちるか?**（一番効く問い。落ちないなら空洞）
- [ ] アサーションがビジネスルール/仕様そのものを検証しているか、単に「エラーが出ない」だけか
- [ ] 異常系・例外パスにテストがあるか（ハッピーパスだけになっていないか）
- [ ] 境界値が入っているか（0件・上限・下限・null・空文字・空リスト・オフバイワン）
- [ ] モックが「外部依存だけ」に限定されているか（テスト対象そのものをモックしていないか）
- [ ] テスト名が「何を検証するか」を表しているか（`test_1`, `test_service` のような無意味な名前は中身も薄い傾向）
- [ ] 検証している validation/ガードが**本番の実行経路に実際に配線されているか**（ユニット単体では弾けても、呼び出し側＝API・Service がその型/検証を通していなければ、そのテストの保証は宙に浮く）。例: schema の `Literal`/`field_validator` を単体テストしていても、エンドポイントが生の `str` を受けていれば無効値は本番で弾けない。「単体で通ることを確認したテスト」を見たら、その検証が**実際に踏まれる経路**を一度たどる。
- [ ] **フィクスチャが「相異なる値」を使っているか**（取り違えを検出できる値か）。複数の同型出力（例: 虚報率と見逃し率）を返す関数で、テストの入力が同値（`fp_num=1, miss_num=1`）だと、実装が両者を**入れ替えても結果が変わらず**すり抜ける。分子・引数・出力先が複数あるなら、それぞれ**別の値**にして「取り違えたら落ちる」状態にする。これは mutation testing でも見えない盲点（演算子変異は代入の入れ替えを作らない）。
- [ ] **同値クラスの抜けが無いか**（0/NULL/正常の3値を別扱いする関数で全クラスを踏むか）。例: 「該当0件→0.0」と「対象なし→NULL」を区別する仕様なら、両方のケースがテストにあるか。片方が無いと「0 のとき None を返す」等の変異が生存する。

## アサーションの強度（アンチパターン集）

生成テストで頻出する「弱いアサーション」。見つけたら補強候補。

| 弱いアサーション | 問題 | 補強の方向 |
|---|---|---|
| `assert result is not None` | 中身を一切見ていない | 期待した具体値・件数・状態を検証 |
| `assert response.status_code == 200` のみ | ボディ・副作用を見ていない | レスポンスボディ、DB状態、発火した副作用を検証 |
| `assert mock.called` のみ | 呼ばれたことだけ。引数・回数を見ていない | `assert_called_once_with(...)` で引数まで |
| `assert len(result) > 0` | 件数も中身も曖昧 | 期待件数の厳密一致 + 要素の内容 |
| `assert True` / アサーション無し | 実質何も検証していない | 空洞。要書き直し |
| 実装から計算した期待値 | 実装が間違うと期待値も間違う（トートロジー） | 期待値はハードコード or 独立に導出 |

## モックの罠

- [ ] テスト対象のクラス/関数自体をモックしていないか（自作自演）
- [ ] モックの戻り値が現実的か（実際のスキーマ・型と一致しているか）
- [ ] モックしすぎてビジネスロジックを一切通していないテストになっていないか
- [ ] 外部I/O（FTP・WebSocket・HTTP・時刻・乱数）は適切にモック/固定されているか（非決定性の排除）

## 層別: 良いテストの姿と赤信号

### ◎ Service / UseCase（最優先）

**良い姿**
- ビジネスルールの分岐が全て網羅されている（各条件のtrue/false）
- 異常系（不正入力・依存先の失敗・競合状態）にテストがある
- モックは外部I/Oだけ。ドメインロジックは実物を通す
- DBが絡むなら transactional ROLLBACK 分離の統合テストで実データ経路を検証

**赤信号**
- 正常系1本だけ
- Service全体をモックして「呼び出しの配線」しか見ていない
- 例外を送出すべきケースのテストが無い
- mutation testing で survived mutants が多い（`references/mutation-testing.md` 参照）

### ✅ Validator / Transformer

**良い姿**
- `@pytest.mark.parametrize` で valid / invalid / 境界を一覧化
- 各バリデーションルールに対し「通るケース」と「弾くケース」が対になっている
- 変換の往復（transform → 逆変換）や不変条件を検証

**赤信号**
- validなケースだけで、invalidを弾けるかのテストが無い
- 境界（最大長・最小値・空・型違い）が抜けている
- Pydanticの `field_validator` / `model_validator` に対応するテストが無い

**parametrizeの型**
```python
@pytest.mark.parametrize("value,valid", [
    ("normal@example.com", True),
    ("", False),           # 空
    ("a" * 256, False),    # 上限超過
    ("no-at-sign", False), # 形式違反
])
def test_email_validation(value, valid):
    ...
```

### ✅ utils

**良い姿**
- 純粋関数として入出力の対応をparametrizeで網羅
- エッジケース（0・負・空・巨大・None）中心

**赤信号**
- 代表値1つだけ
- 例外を投げる入力のテストが無い

### ❌ getter / 純粋DTO

原則テスト不要。ただし以下は「純粋DTO」ではないので Validator 層扱いにする。

- `field_validator` / `model_validator` を持つ Pydantic モデル
- `@computed_field` / `@property` で導出ロジックを持つもの
- カスタムシリアライザ / `model_serializer` を持つもの

これらにロジックがあるのにテストが無い場合は欠落として報告する。

## フロントエンド（Vitest / React）の追加観点

- [ ] ロジック（hooks・純粋関数・状態遷移）と表示（レンダリング）を区別してテストしているか
- [ ] TanStack Query 絡みは loading / error / success の各状態をテストしているか
- [ ] ユーザー操作（クリック・入力）の結果として起きる状態変化を検証しているか（描画されただけで終わっていないか）
- [ ] 非同期の待機（`waitFor` / `findBy`）を適切に使い、フレークになっていないか

## チェックリストの使い方

1. 対象テストファイルを開く
2. 「全層共通」→「アサーション強度」→「モックの罠」→「該当層」の順で当てる
3. 引っかかった項目を、SKILL.md のレポート形式の「検出した空洞・欠落」に優先度付きで転記する
4. 補強を依頼されたら、欠落を埋めるテストを書き、必ず「一度落ちること」を確認する
