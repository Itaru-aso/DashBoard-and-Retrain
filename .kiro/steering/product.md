## 目的 (Purpose)

AI 外観検査システム（検査対象：ジッパー、色ごとのモデルで検査）を対象に、
検査結果のモニタリング・品質劣化の検知・保守タスク管理・モデル再学習とデプロイ・
色ごとの量産検証→実生産移行の管理までを一気通貫で支える Web アプリ。
「検査結果を可視化 → 閾値逸脱を検知してタスク化 → 再学習で是正 → 検査PCへ再デプロイ
→ KPI 達成で実生産へ昇格」という品質改善のクローズドループを提供する。

## 提供価値 (Value Proposition)

- 量産検証・実生産の検査結果を時系列／集計で可視化し、品質状況を即座に把握できる
- NG率・KPI（虚報率・見逃し率）の閾値逸脱を自動検知し保守タスクを自動生成、
  「見逃し・対応漏れ」を防ぎモデル保守のリードタイムを短縮する
- 保守担当者が Web 画面から再学習を実行・監視でき、
  学習〜検査PCへのモデルデプロイを担う（学習用画像は同一PC上の別機能が用意）
- 「未実施 → 量産検証 → 実生産」移行を、検査実績と KPI 達成度に基づき
  自動で判定・進行させ、属人的な移行判断をなくす

## ドメイン定義・用語 (Domain Model & Glossary)

本アプリの中心概念は「色」だが、文脈により指す粒度が異なるため、同一性キーを以下に確定する。
実装はこの粒度を中心にデータモデルを構成すること。

### 同一性キー

- **色マスター（検査対象バリアント）の同一性**: `color_no × size × chain × tape`（4属性タプル）。
  1 レコードが 1 つの検査対象バリアントを表す。1 つの `color_no` に対し
  size / chain / tape 違いで複数レコードを持ちうる。
- **モデルの同一性**: `(monochro／color) × color_no × size × chain × tape`。
  すなわち **1 つの色マスターレコードにつき monochro と color の 2 モデルが対になって対応する**。
- **保守タスクの重複防止キー**: `color_no + size + chain + tape + task_type`。
  色マスター／モデルと同一の物理仕様粒度（`tape` を含む）で重複判定する。

**対象規模**: 総モデル本数 ＝ 対象タプル数 × 2（monochro / color）。
`color_no` は現状約 500 件・増加見込みのため、対象タプル数とモデル本数もこれに連動して増える。

### 用語

- `color_no`: 色番号。人が「色」と呼ぶときの主たる識別子。現状約 500 件で、今後追加され増えていく。
- `size`: ジッパーの番手（想定値: `03` / `05` / `45`）。
- `chain`: チェーン種別（想定値: `CZT8` / `CZT10`）。
- `tape`: テープ種別。**基本的に空白**（値が入るのは例外的）。空白でも同一性キーの一部として扱う。
- `monochro` / `color`: 1 つの検査対象バリアントに対し常に対で並走する 2 系統のモデル。
- 色見本: 色の基準色情報（RGB／Lab）。

## 不変条件・業務ルール (Invariants & Business Rules)

エージェントが実装中に**絶対に破ってはならない**制約。受け入れ基準（spec 層）の起点とする。

1. **モデルは常に対**: 1 つの検査対象バリアントには monochro / color の 2 モデルが対応する。
2. **タスク重複防止**: 同一キー `color_no + size + chain + tape + task_type` で重複起票しない。
   同キーの既存タスクが **OPEN なら上書き、IN_PROGRESS / DONE なら保持**（上書きしない）。
3. **タスク状態遷移**: `OPEN → IN_PROGRESS → DONE` の前進方向のみ。
4. **色ライフサイクル遷移（一方向・後戻り不可）**: `未実施 → 量産検証 → 実生産`。逆遷移は発生しない。
   - 「量産検証」へ: 検査実績の発生で自動遷移。
   - 「実生産」へ: **日単位（JST 日次）**で `虚報率 ≤ 1.5%` かつ `見逃し率 ≤ 0.05%` を**同時達成**した日が
     **1 日でも発生した時点で即時に自動昇格**（連続達成は不要、人手承認なし）。判定母数はその日の生産数。
     各率はその日の実測比を**絶対値**としてそのまま基準値と比較する（最小サンプル数の下限は設けない）。
     ※ 評価粒度は他機能（ダッシュボード・逸脱判定）と統一して**日次**とする（ロット概念は不使用）。
5. **再学習ジョブ状態遷移**: `QUEUED → RUNNING → COMPLETED / FAILED / CANCELLED`。
6. **閾値の優先順位**: `per_color > global`（色別が全体に優先）。閾値は有効期間を持つ。

## 主要機能 (Core Capabilities)

- **検索・ダッシュボード**: 期間／色／サイズ／チェーン等で検査記録を絞り込み、
  生産実績（スループット）推移・NG率推移・量産検証 KPI（虚報率・見逃し率）を
  グラフ／表で可視化（閾値ラインの重ね描き対応）
- **タスク自動生成・管理**: NG率・KPI の閾値逸脱で保守タスクを自動起票し
  （重複防止・状態遷移は「不変条件」2・3 に従う）、コメント・再発防止メモを更新できる
- **色マスター・色ライフサイクル管理**: 色マスターレコードの登録と色見本（RGB／Lab）管理、
  ステータスの自動遷移判定（「不変条件」4 に従う）
- **モデル再学習ワークフロー**: 再学習ジョブのキュー管理（「不変条件」5）、
  学習パイプラインの実行、ジョブの進捗・ログのリアルタイム表示、
  monochro／color 2 モデルの学習〜検査PCへのデプロイ
- **エッジPC（検査PC）管理**: **モデル配信先**となる接続先の登録・管理（学習用画像の収集元ではない）
- **閾値管理**: メトリクス別・色別／グローバル、有効期間付き（「不変条件」6）
- **検査結果ダッシュボード／日次集計**: 取り込まれた検査データに対する JST 日次の集計・KPI 可視化（NG率・虚報率・見逃し率・スループット）

## 確定済みの外部制約 (Fixed Environmental Constraints)

ver1 から引き継ぐ、本アプリ側では選択できない環境制約。技術スタックの詳細選定は `tech.md`。

- **オンプレ前提**: LAN 内利用。インターネット経由の外部公開はしない。
- **検査PC（エッジPC）連携**: FTP で連携（**モデル配信**：アプリ→エッジ）。
  学習用画像は同一PC上の別機能が用意するため**FTP 収集は行わない**。
- **エッジ側モデル形式**: ONNX。
- **モデル構成**: monochro / color の 2 モデル系統。
- ※ リアルタイム配信方式・学習ジョブ実行方式・フレームワーク・言語・DB 等の選定は `tech.md` で行う。

## 対象ユーザー (Target Users)

- **作業者**: 検査状況・NG率の日次モニタリング
- **保守担当者**: モデル保守、閾値チューニング、トラブル対応、保守タスク管理、
  色ライフサイクルの管理、および再学習の実行・監視

アプリ内ユーザー管理は最小限。

## スコープ外 (Out of Scope)

- 外観検査 AI の**モデルアーキテクチャの新規研究開発**（既存モデルの再学習は対象に含む）
- 画像データの恒久保存・解析（再学習時に同一PC上の所定パスから利用する画像は除く。検査結果は DB のメタ情報で扱う）
- ログファイル取り込み処理そのもの（別プロセスで実装済み。本アプリは取り込み結果 DB を対象とする）
- **取り込み状況の可視化**（ファイル単位の取り込み状態・件数・エラー一覧）。検査PCからの取り込み有無の確認は
  本アプリの役割ではなく、取り込み台帳データも DB に存在しないため**スコープ外**。本アプリは**取り込まれた後のデータ**を扱う。
- インターネット経由の外部公開、スマホアプリ
