# 保守タスク自動生成・管理 — Requirements

> spec: `保守タスク自動生成・管理 (maintenance-task)`
> 配置想定: `.kiro/specs/maintenance-task/requirements.md`
> 前提 steering: `product.md`（主要機能「タスク自動生成・管理」, 不変条件2・3）, `tech.md`, `structure.md`
> 依存 spec: `閾値管理`（`resolve_effective`）／ `検査結果ダッシュボード`（メトリクス算出ロジックを共有）

## 概要 (Introduction)

メトリクス（NG率・虚報率・見逃し率）を有効閾値と比較して**逸脱を検知**し、逸脱時に**保守タスクを自動起票**する。
タスクは重複防止のうえ upsert し、ステータス遷移・コメント・再発防止メモで管理する。
タスクテーブルは **ver2 DB**（自前・Alembic 管理）に置く。

### スコープ (In Scope)
- 逸脱判定（メトリクス算出 ＋ `resolve_effective` による有効閾値との比較）。
- 逸脱時の保守タスク自動起票（重複防止・upsert）。
- タスク管理: ステータス遷移（前進のみ）・コメント・再発防止メモ・一覧/参照。

### スコープ外 (Out of Scope)
- メトリクスの算出ロジック自体（`検査結果ダッシュボード` と共有。重複実装しない）。閾値の管理（`閾値管理`）。
- 取り込み（外部・スコープ外）。
- 手動でのタスク作成（**自動起票のみ**。将来必要なら `task_type` に手動種別を足して非破壊で追加）。

### 用語・前提
- 評価単位: `日次 × color_no × size × chain × tape × メトリクス`（メトリクス定義はダッシュボード spec に準拠）。
- 逸脱: メトリクス値が有効閾値を**超える**（`値 > 閾値`。すべて上限値）。
- 不変条件（`product.md`）: 重複防止キー `color_no + size + chain + tape + task_type` ／
  ステータス遷移 `OPEN → IN_PROGRESS → DONE` の前進のみ。

## 要件 (Requirements)

### R1. 逸脱判定
**User Story**: 保守担当者として、品質が基準を超えたら自動で気づきたい。

**受け入れ基準 (EARS)**
1. 評価単位ごとに、メトリクス値と `resolve_effective(metric, fulltuple, 日)` の有効閾値を比較し、
   `値 > 閾値` のとき**逸脱**と判定する（SHALL）。
2. 有効閾値が無い（解決が「閾値なし」）単位は逸脱判定しない（SHALL。`閾値管理` の下流契約）。
3. KPI（虚報率・見逃し率）がラベル0件で算出不能（NULL）の単位は判定をスキップし、WARN ログを出す
   （SHALL。`calculate_ngrate_KPI.md`）。
4. 逸脱判定は**日次スケジュールで定期実行**する（**アプリ内スケジューラ**で日次ジョブ。Docker コンテナ＝Linux 上で
   dev/prod とも同一に動作）。直近の一定期間を再評価し、R2 の upsert で**冪等**に反映する（実行漏れ・遅延データも
   次回実行で取り戻せる）。タスクの**自動クローズは行わない**（クローズは R3 の手動遷移）。
   任意で手動「いま評価」エンドポイントを併設してよい。

### R2. タスク自動起票（重複防止・upsert）
**User Story**: 同じ問題で重複タスクが乱立せず、対応中のタスクが勝手に書き換わらないでほしい。

**受け入れ基準 (EARS)**
1. 逸脱を検知したとき、重複防止キー `color_no + size + chain + tape + task_type` で
   **アクティブ（`OPEN`/`IN_PROGRESS`）なタスク**を探す（SHALL）。
2. アクティブなタスクが無ければ（皆無、または過去の `DONE` のみ）、`OPEN` の新規タスクを作成する
   （SHALL。過去の `DONE` は履歴として保持＝**再発時は再起票**）。
3. 既存の `OPEN` がある場合、**上書き**（最新の逸脱内容で更新）する（SHALL）。
4. 既存の `IN_PROGRESS` がある場合、**保持**（上書きせず新規も作らない。対応中のため）する（SHALL）。
5. 重複防止は「同キーにアクティブ（`OPEN`/`IN_PROGRESS`）なタスクは高々1件」を保証する（SHALL）。
6. `task_type` は逸脱したメトリクスに対応する（`ng_rate` / `false_alarm_rate` / `miss_rate` の3種）。

### R3. ステータス遷移
**受け入れ基準 (EARS)**
1. ステータスは `OPEN → IN_PROGRESS → DONE` の前進方向のみ遷移できる（SHALL。`product.md` 不変条件3）。
2. 逆遷移・段飛ばしの要求は拒否する（SHALL）。

### R4. コメント・再発防止メモ
**受け入れ基準 (EARS)**
1. タスクにコメントを追加できる（SHALL）。
2. タスクの再発防止メモを更新できる（SHALL）。

### R5. 一覧・参照
**受け入れ基準 (EARS)**
1. タスク一覧をステータス・色・`task_type`・期間で絞り込んで取得できる（SHALL）。

### R6. アクセス
**受け入れ基準 (EARS)**
1. 認証は Basic 認証（ロール制御なし）。ログイン済みユーザが操作できる（SHALL）。
2. 各タスクは作成・更新の日時を保持する（SHALL）。

## 確定事項 (Resolved)

- 評価は**日次粒度**・**日次スケジュール定期実行**（アプリ内スケジューラ）・**冪等**・自動クローズ無し。
- 重複防止: 同キーにアクティブ（`OPEN`/`IN_PROGRESS`）なタスクは高々1件。`DONE` 後の再逸脱は新規起票（再発）。
- `task_type` は逸脱メトリクス（`ng_rate`/`false_alarm_rate`/`miss_rate`）。起票は**自動のみ**（手動作成は当面なし）。

## design への申し送り (Notes for Design)

- ver2 DB のタスクテーブル（補填・Alembic）。「同キーにアクティブ1件」を**部分ユニーク制約**
  （`WHERE status IN ('OPEN','IN_PROGRESS')`）で DB レベルに担保。
- 逸脱判定の評価サービス: 日次メトリクス算出は `検査結果ダッシュボード` と共有し、`resolve_effective` と比較。
- アプリ内スケジューラ（日次ジョブ・冪等・直近一定期間を再評価）。**`tech.md`／`structure.md` に定期処理を反映**。
- 実行環境は Docker（Linux コンテナ）で dev/prod 同一 ── `tech.md` の「Docker なし Windows 対応」記述の整理も併せて検討。
