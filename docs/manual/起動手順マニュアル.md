# shisui app_ver2 起動手順マニュアル

YKK ジッパー外観検査 AI モデル管理 Web アプリの**ローカル起動手順**をまとめる。
対象読者：本アプリを自分の PC / サーバで動かす担当者。

> 用語・規約の正は [`CLAUDE.md`](../../CLAUDE.md)、仕様は [`.kiro/specs/`](../../.kiro/specs/)。本書は「動かす」ことに絞る。

---

## 0. 全体像

本アプリは 3 つの構成要素からなる。

| 要素 | 役割 | 既定ポート |
|---|---|---|
| **ver2 DB**（PostgreSQL） | 自前 DB（読み書き・Alembic 管理） | 5432 |
| **業者検査 DB / app_db**（PostgreSQL） | 検査明細（**読み取り専用**） | 5433 |
| **backend**（FastAPI + Uvicorn） | API・日次スケジューラ・再学習キュー | 8000 |
| **frontend**（React + Vite） | 画面（開発時は devserver・`/api` を backend へプロキシ） | 5173 |

起動方法は 2 通り。目的に合わせて選ぶ。

- **A. Docker で起動**（DB＋backend を一括・手早く動かす） → [第2章](#2-a-docker-で起動する推奨クイックスタート)
- **B. ローカル開発で起動**（venv／npm・コード変更しながら開発する） → [第3章](#3-b-ローカル開発で起動する)

前提ソフトウェア：
- Docker Desktop（方法 A）
- Python 3.11 / PostgreSQL 14 / Node.js 18 以上（方法 B）

## 1. 環境変数の準備（両方式で共通）

バックエンドは `.env` から設定を読む。サンプルをコピーして作成する。

```bash
cd backend
cp .env.example .env      # PowerShell: Copy-Item .env.example .env
```

主な変数（詳細は `.env.example` と [`CLAUDE.md`](../../CLAUDE.md) の「設定」）：

| 変数 | 説明 | 例 |
|---|---|---|
| `DATABASE_URL` | ver2 DB（読み書き・必須） | `postgresql+psycopg2://shisui:shisui@localhost:5432/ver2` |
| `INSPECTION_DATABASE_URL` | app_db（読み取り専用・必須） | `postgresql+psycopg2://readonly:readonly@localhost:5433/app_db` |
| `ENVIRONMENT` | `development` / `production` | `development` |
| `LOG_LEVEL` | ログレベル | `INFO` |
| `ENABLE_BASIC_AUTH` | Basic 認証の有効化（有効時 `BASIC_AUTH_PASS` 必須） | `false` |
| `BASIC_AUTH_USER` / `BASIC_AUTH_PASS` | 単一共有クレデンシャル | `shisui` / （任意） |
| `AGG_RUN_TIME` / `AGG_WINDOW_DAYS` | 日次集計ジョブの実行時刻・再集計窓 | `02:00` / `7` |
| `BREACH_EVAL_ENABLED` / `BREACH_EVAL_TIME` | 逸脱判定ジョブの有効化・時刻 | `false` / `03:00` |
| `TRAINING_DIR` / `TRAINING_MODEL_DIR` / `TRAINING_PYTHON` | 再学習（`training/` 連携。[第5章](#5-再学習training-連携)） | 環境に応じて |

> `.env` は**コミットしない**（`.gitignore` 済み）。シークレットを含めてよい。

## 2. （A）Docker で起動する（推奨・クイックスタート）

DB 2 台と backend をまとめて立ち上げる。`docker-compose.yml` が環境変数を内包するため、この方式では `.env` は不要（コンテナ内で設定される）。

### 2.1 起動

```bash
# リポジトリルートで実行
docker compose up -d --build
```

- `ver2-db`（postgres:14・DB=`ver2` / user=`shisui`）
- `inspection-db`（postgres:14・app_db 代役・空）
- `backend`（`http://localhost:8000` で公開）

が起動する。DB はヘルスチェック通過後に backend が起動する。

### 2.2 DB マイグレーション（初回・スキーマ更新時に必須）

コンテナ起動だけでは**テーブルは作られない**。ver2 DB にマイグレーションを適用する。

```bash
docker compose exec backend alembic upgrade head
```

> app_db（業者検査 DB）は Alembic 対象外（読み取り専用）。

### 2.3 動作確認

```bash
curl http://localhost:8000/health
# {"status":"healthy","ver2_db":"ok","inspection_db":"ok"} が返れば OK
```

- API ドキュメント（Swagger UI）：`http://localhost:8000/docs`

### 2.4 停止

```bash
docker compose down          # コンテナ停止（データは保持）
docker compose down -v       # ボリュームごと削除（DB を初期化）
```

### 2.5 フロントエンドについて（重要な注意）

現行の `docker-compose.yml` / `Dockerfile` は **backend のみ**を配信する。画面を見るには、方法 B の手順で **フロントを別途起動**する（[3.4](#34-フロントエンドの起動)）。

- 開発：`npm run dev`（`/api` は `localhost:8000` の backend へプロキシ）
- 本番一体配信（`ENVIRONMENT=production` で backend が `frontend/dist` を配信）は、**現状の backend イメージに `frontend/dist` が含まれていない**ため未対応。将来対応（[第6章](#6-既知の制限と次のステップ)）。

## 3. （B）ローカル開発で起動する

コードを変更しながら開発する場合の手順。

### 3.1 PostgreSQL を用意

ver2 用（5432）と app_db 用（5433）の 2 つが必要。手元に無ければ Docker で DB だけ立てるのが簡単。

```bash
# ver2 DB（読み書き）
docker run -d --name ver2-db -p 5432:5432 \
  -e POSTGRES_USER=shisui -e POSTGRES_PASSWORD=shisui -e POSTGRES_DB=ver2 postgres:14

# app_db（業者検査 DB 代役・読み取り専用想定）
docker run -d --name app-db -p 5433:5432 \
  -e POSTGRES_USER=readonly -e POSTGRES_PASSWORD=readonly -e POSTGRES_DB=app_db postgres:14
```

`.env` の `DATABASE_URL` / `INSPECTION_DATABASE_URL` を上記に合わせる（`.env.example` の既定値がこの構成）。

> `docker compose` の DB サービスはホストにポート公開していないため、ローカル backend からは上記の単体コンテナ（ポート公開あり）か、自前の PostgreSQL を使う。

### 3.2 バックエンドのセットアップ

```bash
cd backend
python -m venv .venv
# 有効化: Windows(PowerShell) → .venv\Scripts\Activate.ps1 / macOS・Linux → source .venv/bin/activate
pip install --upgrade pip
pip install -e .            # pyproject.toml の依存＋src パッケージを導入
```

### 3.3 マイグレーション＆バックエンド起動

```bash
# ver2 DB にスキーマを適用（app_db は対象外）
alembic upgrade head

# 起動（単一ワーカ必須＝スケジューラ・再学習キューの単一所有）
uvicorn src.main:app --host 0.0.0.0 --port 8000 --workers 1 --reload
```

> **`--workers 1` は厳守**。複数ワーカにするとスケジューラ・再学習キューが多重化する。

確認：`http://localhost:8000/health` / `http://localhost:8000/docs`

### 3.4 フロントエンドの起動

別ターミナルで：

```bash
cd frontend
npm install
npm run dev                 # http://localhost:5173（/api は backend:8000 へプロキシ）
```

ブラウザで `http://localhost:5173` を開く。主な画面：

| パス | 画面 |
|---|---|
| `/dashboard` | ダッシュボード（3指標 KPI） |
| `/tasks` | 保守タスク管理 |
| `/thresholds` | 閾値管理 |
| `/colors` | 色マスター |
| `/edge-pcs` | エッジPC（検査PC）管理 |
| `/retraining` | モデル再学習（ライブ進捗・配信） |

> 現状トップ（`/`）は見出しのみでナビゲーションメニューは未実装。各画面へは上記 URL を直接開く。

### 3.5 本番ビルド（任意）

```bash
cd frontend
npm run build               # frontend/dist を生成
```

`ENVIRONMENT=production` かつ `frontend/dist` が存在すれば、backend が同一オリジンで SPA を配信する（未知パスは `index.html` にフォールバック）。ローカルで一体起動する場合はこの後 backend を `ENVIRONMENT=production` で起動する。

## 4. 検証・トラブルシュート

### 4.1 テスト（動作確認）

```bash
# バックエンド（カバレッジゲート cov≥80）
cd backend && pytest

# フロントエンド
cd frontend && npm run test
```

### 4.2 よくあるつまずき

| 症状 | 原因 / 対処 |
|---|---|
| `/health` が `503` / `ver2_db: error` | ver2 DB に接続できない。`DATABASE_URL` と DB の起動・ポートを確認 |
| `inspection_db: unavailable`（`status` は healthy） | app_db 未接続。**参考扱い**で致命ではない。明細表示を使う場合のみ要接続 |
| 起動時に `ValidationError`（`DATABASE_URL` 等） | `.env` 未作成/必須変数未設定。[第1章](#1-環境変数の準備両方式で共通)参照 |
| テーブルが無い / API が 500 | マイグレーション未適用。`alembic upgrade head` を実行 |
| 画面から API が 404/接続不可 | backend（:8000）未起動、または Vite プロキシ不整合。backend を先に起動 |
| 401 が返る | `ENABLE_BASIC_AUTH=true`。ユーザ/パスを付与するか開発中は `false` に |

## 5. 再学習（`training/` 連携）

`/retraining` 画面からの再学習は、`training/pipline.py` を **subprocess** で起動する（学習ロジックは本アプリでは変更しない）。実行には別途以下が必要：

- `training/` に学習パイプライン一式（配置済み）＋ **教師重み `training/0_pretraining/*.pth`**（`.gitignore` のためデプロイ時に配置）
- 学習用 Python 環境（PyTorch/CUDA）と **GPU**（本番は 2 枚想定）
- `.env` の `TRAINING_DIR`（＝`training/` の絶対パス）/ `TRAINING_MODEL_DIR`（ONNX 出力ルート）/ `TRAINING_PYTHON`（学習環境の python）

ver2 連携時は `common.skip_download=true` / `common.skip_upload=true` で学習側の FTP を無効化し、配信は ver2 の `deployment_service`（有効エッジPC全台へ FTP 送信）が担う。

> GPU/CUDA を含むランタイムは現行 backend イメージ（`python:3.11-slim`）には**未同梱**。再学習を動かすには GPU 対応環境が別途必要（[第6章](#6-既知の制限と次のステップ)）。

## 6. 既知の制限と次のステップ

- **フロントの本番一体配信**：現 backend イメージに `frontend/dist` が含まれず、Docker では SPA を配信しない。運用時は「フロントを別配信」または「`frontend/dist` を含めた backend イメージ化」が必要。
- **GPU/CUDA ランタイム**：`Dockerfile` は骨組み（`python:3.11-slim`・torch 非同梱）。再学習を動かすには base を `nvidia/cuda` 系へ差し替え、`docker-compose` の `deploy.resources` で GPU を渡す（コメントに雛形あり）。
- **業者検査 DB（app_db）**：Docker の `inspection-db` は空の代役。実データは別途 dump 投入が必要（明細・KPI 集計に必要）。
- **ナビゲーションメニュー**：未実装（各画面は URL 直接遷移）。
