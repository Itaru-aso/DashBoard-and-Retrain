# モデル再学習ワークフロー — 既存 `training/` 連携のための情報要件

> 目的: 既に実装済みの**モデル学習ロジック（`training/` パイプライン）**を、`モデル再学習ワークフロー` spec
> （ジョブ管理・キュー・進捗配信・配信・現行モデル記録）に噛み合わせて実装するために必要な情報を整理する。
> spec 側の想定（`from pipline import`・`subprocess` 起動・2GPU・ONNX 出力・FTP 配信）と実物のズレを埋めるための確認項目。
>
> 最も効率が良いのは `training/` の実物（エントリポイント・引数パース・出力保存・進捗ログ・FTP 配信箇所・README・設定サンプル）を共有すること。
> コードがあれば下記の多くは読み取れるため、不明点のみ回答すればよい。

---

## A. 最優先（骨格が決まる・これが無いと subprocess 起動部を書けない）

### A-1. 起動インターフェース
- 起動方法はどちらか:
  - **コマンドライン**: 実際のコマンド全文（例 `python -m training.train` / `python training/run.py ...`）。
  - **関数 import**: モジュールと関数（例 `from pipline import run_training`）とそのシグネチャ。
- 想定（`structure.md`）は `from pipline import`（綴りは意図的に `pipline`）。実物の綴り・パスを確認。

### A-2. 入力パラメータ（引数名・型・例つき）
- **対象色の渡し方**: `color_no` 単体／フルタプル `color_no・size・chain・tape`／データセット ID のいずれか。
- **学習用画像の所在**: パス引数（例 `--data-dir`）か、固定パスか。
- **monochro / color の扱い**: 1回の起動で両方学習するのか、別々に2回起動するのか。
- **GPU 指定**: 引数（例 `--gpus 0,1`）／環境変数 `CUDA_VISIBLE_DEVICES`／プログラム任せ、のどれか。
- 実コマンド例が1つあると一気に確定する。例:
  `python training/run.py --color-no 501 --size 05 --chain CZT8 --tape "" --data-dir /data/501 --gpus 0,1 --out /models/501`

### A-3. 出力（成果物）
- 学習完了時に **何が・どこに・どんなファイル名で** 出るか（monochro/color の ONNX それぞれ）。
  例: `/models/501/501_color_model.onnx`・`/models/501/501_mono_model.onnx`。
- **成功/失敗の判定方法**: プロセス終了コード 0／特定ファイルの生成／標準出力の完了マーカー、のどれか。
- → `retraining_job` に記録するパス・成功判定の根拠になる。

## B. 重要（挙動の質・例外系を決める）

### B-1. 進捗・ログの出力
- 学習中の進捗/ログの出力先: **標準出力/標準エラー**か、ファイルか。
- 形式: 行テキスト（例 `epoch 3/50 loss=0.12`）／JSON 行／進捗率の数値、のどれか。
- → WebSocket 進捗配信で**そのまま流す**か**整形して流す**かを決める。

### B-2. 失敗時の挙動
- 失敗時の終了コード・エラーメッセージの出方。
- 途中失敗時に**中途半端な ONNX が残る**か。
- 同じ出力先への**再実行（上書き）が安全**か。
- → ジョブを `FAILED` にする条件・リトライ・キャンセル設計に直結。

### B-3. 実行時間・リソース・同時実行
- 1ジョブの目安時間（数分／数時間）。
- **GPU 2枚を占有**するか。
- **同時複数実行の不可**などプログラム側の制約（spec は「同時1本・FIFO」想定。矛盾の有無を確認）。

### B-4. キャンセル
- 学習を**途中で止められる**か。プロセス `kill` で問題ないか（GPU メモリ・ロックが残らないか）。
- **子プロセスを生む**か（プロセスツリーごと kill が必要か）。
- → キャンセル機能の実装可否・方式に関わる。

## C. 配信まわり（学習と配信の境界を決める）

### C-1. ONNX の FTP 配信
- 検査PC への FTP 配信を、**既存プログラムが学習完了時に自動でやる**のか、**ver2 側（`deployment_service`）で実装**するのか。
- spec の方針: **FTP の実 I/O は `training/` 側**、**配信トリガー・現行モデル記録は ver2**。
- 既存に配信処理がある場合は、その**呼び出し方（関数・引数）**。配信先は `エッジPC管理` の `model_port`（有効な全台へ配信）。

## D. 環境・前提

### D-1. 学習用画像の用意
- 画像を**誰が・いつ・どこに**置くか（spec: 同一PC上の別機能が所定パスに用意・収集はスコープ外）。
- 再学習起動時点で**画像が揃っている前提**でよいか。

### D-2. 依存・実行環境
- `training/` が要求する **Python バージョン・主要ライブラリ（torch 等）・CUDA**。
- → 確定済み Docker イメージ（`nvidia/cuda` 12.8+ ＋ Python 3.11 ＋ PyTorch cu128 系・Blackwell 対応）と整合するか確認。

### D-3. 設定ファイル
- 引数でなく**設定ファイル（YAML/JSON）**で動く場合、そのサンプルと、**ジョブごとに差し替える項目**。

## まとめ（提出物の優先度）

1. **最優先**: `training/` のエントリポイント＋引数まわり（= A-1/A-2/A-3）。
   - 最低限: **実際の起動コマンド1行** ＋ **出力 ONNX のパス・ファイル名規約** ＋ **成功判定方法**。
2. 次点: 進捗ログの形式（B-1）・失敗時/キャンセル（B-2/B-4）・配信の担当（C-1）。
